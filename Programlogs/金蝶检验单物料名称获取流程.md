# 金蝶 QM_InspectBill 检验单"物料名称"获取全流程

**整理时间**: 2026-07-16
**背景**: 检验单列表卡片（`screens/order-list/index.tsx`）只显示单据编号，不显示物料名称，用户希望在单据编号后追加物料名称方便识别。本文档记录从"确认数据能否取到"到"确定正确字段写法"再到"落地展示"的完整调试过程。

---

## 一、整体链路概览

```
账号密码/签名登录 (LoginBySign)
        │  拿到 Set-Cookie: kdservice-sessionid=xxx
        ▼
BillQuery 查询 QM_InspectBill
  FieldKeys 中加入 FMaterialId.FNumber / FMaterialId.FName
        │  返回按分录展开的对象数组（一单可能对应多行）
        ▼
按 FID 聚合分录行 → 取首个物料 + 统计物料总数
        │
        ▼
OrderSummary.material_name / material_count
        │
        ▼
列表卡片：单据编号同行内联展示「单号 · 物料名称 等N种」
```

---

## 二、第一步：验证登录 + BillQuery 链路是否可用

在改动任何展示逻辑之前，先写了一个独立 Node 脚本 `scripts/verify-billquery.js`，脱离 App 直接对金蝶真实服务器验证登录和查询，避免在 UI 里调试时无法区分"网络问题"和"字段问题"。

脚本复现了 App 内 `api/kingdee/auth.ts` 的 `loginBySign` 签名算法（`SHA256(acctId+userName+appId+appSecret+timestamp)`，数组按 Unicode 码点排序后拼接）和 `api/kingdee/client.ts` 的 `callKingdeePost('BillQuery', ...)` 调用格式。

### 踩坑：登录成功但查询报"会话信息已丢失"

第一次运行时登录响应 `LoginResultType: 1`（成功），但紧接着的 `BillQuery` 请求报错：

```json
{"Result":{"ResponseStatus":{"ErrorCode":500,"IsSuccess":false,"Errors":[{"Message":"会话信息已丢失，请重新登录","DIndex":0}],"MsgCode":1}}}
```

排查发现：登录响应的 HTTP 头 `Set-Cookie` 是 `kdservice-sessionid=xxx`，但 `api/kingdee/auth.ts` 中如果响应体里带有 `KDSVCSessionId` 字段，会用 `KDSVCSessionId=xxx` **覆盖**掉正确的 Cookie 名，导致后续请求携带了服务器不认识的 Cookie 名。

**结论**：验证脚本改为只用 `Set-Cookie` 头解析出的原始 Cookie（`kdservice-sessionid=xxx`），登录后即可正常查询。

> 这是 App 现有代码里的一个真实 bug（`loginBySign`/`validateUser` 都有同样的覆盖逻辑），已告知但按要求暂未修复，不在本次改动范围内。

## 三、第二步：确认 BillQuery 能拿到检验单基本字段

用验证脚本的 `LIST_FIELD_KEYS`（`FID,FBILLNO,FDate,FBILLTYPEID.FNumber,FBILLTYPEID.FName,FDocumentStatus`）查询最近 90 天数据，登录+查询链路打通，拿到真实检验单列表（IQC/FQC/RQC/INQC 等编号）。

在此基础上做了一次全量导出（不限日期），验证共 120 条单据、按类型和状态统计，生成了 `Programlogs/金蝶检验单全量数据导出.md`。此时导出报表**没有物料信息**——因为 `FieldKeys` 本来就没有查这两个字段。

## 四、第三步：探索物料字段的正确写法（重点踩坑记录）

### 4.1 最初的错误假设：加 `FEntity_` 前缀

参考金蝶官方文档中"查询单据体内码需加单据体 Key 和下划线，如 `FEntryKey_FEntryId`"的说明，最初以为物料字段也要写成：

```
FEntity_FMaterialId.FNumber
FEntity_FMaterialId.FName
```

实测全部报错：

```json
{"Message":"元数据中标识为FEntity_FMaterialId的字段不存在"}
```

尝试了十几种大小写/命名变体（`FEntity_FMATERIALID`、`FEntity_FMaterialId_Id`、`FEntity_MaterialId` 等）均失败，只有分录内码字段 `FEntity_FEntryID` 能成功返回值：

```json
[{"FID":100009,"FBILLNO":"IQC000003","FEntity.FEntryID":100009}]
```

（注意返回结果里键名会变成点号 `FEntity.FEntryID`，但**请求**时仍要写下划线 `FEntity_FEntryID`。）

### 4.2 用 View 接口反查真实字段名

为了搞清楚物料字段到底叫什么，改用 `View` 接口直接查看一条真实单据（`Id=100009`）的完整结构，发现：

- 单据体数组的 key 叫 `Entity`（不带 F 前缀，这是 View 接口自己的返回风格，与 BillQuery 无关）
- 分录里物料字段直接叫 `MaterialId`（不带 `FEntity_` 前缀）：
  ```json
  "MaterialId": {
    "Id": 1175636,
    "Number": "12.20.08.03.0340",
    "Name": [{ "Key": 2052, "Value": "家居舱夹胶玻璃/平顶" }],
    ...
  }
  ```
- 检验数量叫 `InspectQty`，单位叫 `UnitId`

### 4.3 反推出 BillQuery 的正确写法

对照单据头字段（`FBILLTYPEID.FNumber` 不需要任何单据体前缀就能取到值）的经验，推断分录字段在 BillQuery 里也**不需要 `FEntity_` 前缀**，直接用字段自身的 Key 即可，服务端会自动定位到单据体：

```
FMaterialId            → 物料内码（如 1175636）
FMaterialId.FNumber    → 物料编码（如 "12.20.08.03.0340"）
FMaterialId.FName      → 物料名称（如 "家居舱夹胶玻璃/平顶"）
FInspectQty            → 检验数量
FUnitID.FNumber        → 单位编码
```

实测全部成功：

```json
[{"FID":100009,"FBILLNO":"IQC000003","FMaterialId.FNumber":"12.20.08.03.0340"}]
[{"FID":100009,"FBILLNO":"IQC000003","FMaterialId.FName":"家居舱夹胶玻璃/平顶"}]
[{"FID":100009,"FBILLNO":"IQC000003","FInspectQty":30.0000000000}]
[{"FID":100009,"FBILLNO":"IQC000003","FUnitID.FNumber":"Pcs"}]
```

### 4.4 原理小结

金蝶 BillQuery 支持在 `FieldKeys` 里用 `字段名.关联字段` 语法跨表取值：`FMaterialId` 本身只是指向物料档案表（`BD_MATERIAL`）的内码引用，加上 `.FNumber`/`.FName` 后，服务端自动做一次关联表 JOIN，把物料档案里对应属性取出来拼进返回结果。这个语法对**单据体分录字段同样生效**，不需要额外加单据体前缀——`FEntryKey_FEntryId` 这种下划线写法只适用于**分录自身的内码**（如 `FEntity_FEntryID`），不适用于分录下的业务字段。

## 五、第四步：全量导出验证 + 一单多物料的发现

用修正后的字段重新跑全量导出脚本 `scripts/export-all-inspectbills.js`，物料信息正确出现：

```
| 1 | 100009 | IQC000003 | ... | 12.20.08.03.0340 | 家居舱夹胶玻璃/平顶 | 30 | Pcs |
```

同时发现：**BillQuery 返回的是"分录行"，不是"单据行"**。例如单据 `IQC000005`（FID=100011）在服务器上有 2 个物料分录，会返回 2 行，行内 `FID`/`FBILLNO` 等主表字段相同，只有物料字段不同：

```json
[
  { "FID": 100011, "FBILLNO": "IQC000005", "FMaterialId.FName": "密封胶泥" },
  { "FID": 100011, "FBILLNO": "IQC000005", "FMaterialId.FName": "硅酮耐候密封胶玻璃胶" }
]
```

全量导出报表原本按行数统计（120 条单据 → 展开成 122 行分录），印证了这一点。

## 六、第五步：落地到列表页展示

### 6.1 数据层改动（`api/kingdee/inspect.ts`）

- `LIST_FIELD_KEYS` 新增 `FMaterialId.FNumber`、`FMaterialId.FName`
- `queryInspectBills()` 的解析逻辑从"逐行直接转 `OrderSummary`"改为**按 FID 聚合**：
  - 同一 FID 只保留第一次出现的物料编码/名称
  - 用一个 `materialCountMap` 统计每个 FID 对应的物料分录行数
  - 最终每个 FID 只输出一条 `OrderSummary`，附带 `material_count`

### 6.2 类型扩展（`api/kingdee/types.ts`）

`OrderSummary` 新增：
```ts
material_count?: number; // 单据体分录（物料）总数，用于列表页展示"等N种"
```

### 6.3 UI 层改动（`screens/order-list/index.tsx`）

- 本地 `InspectionOrder` 接口同步加 `material_count`
- 卡片渲染改为单据编号**同一行内联**展示物料名（此前的实现是物料名单独起一行，一直未生效是因为数据源没有填值）：
  - 单物料：`IQC000035 · 家居舱夹胶玻璃/平顶`
  - 多物料：`IQC000005 · 密封胶泥 等2种`
- 用单个 `<Text numberOfLines={1}>` 包裹两段内联 `<Text>` 子元素（RN 富文本写法），保证整行超长时统一截断，不会把右侧状态胶囊挤出屏幕
- 无物料信息时不显示多余内容，保持原样

### 6.4 验证方式

- `npx tsc --noEmit` 类型检查通过
- 独立 Node 脚本对真实金蝶服务器发起请求验证字段返回值符合预期（含多物料聚合场景 IQC000005 的抽样验证）

## 八、可复用结论（供后续查询其他单据体字段参考）

1. **BillQuery 查询单据体业务字段**：不需要 `FEntity_` 前缀，直接写字段 Key（如 `FMaterialId`、`FInspectQty`），跨表关联属性用点号（如 `FMaterialId.FName`）。
2. **`FEntity_` 前缀仅用于分录自身内码**：如 `FEntity_FEntryID`，不适用于分录下的业务字段。
3. **BillQuery 按分录行返回**：一张单据有 N 个分录就返回 N 行，主表字段重复，业务层需按 `FID` 自行聚合去重。
4. **不同接口字段名风格不同**：
   - BillQuery 请求字段：`FMaterialId.FNumber`（驼峰 + F前缀，点号访问关联属性）
   - View 接口返回字段：`MaterialId`（可能无 F 前缀，风格因表单而异，需要实测确认，不能凭猜测硬编码）
5. **排查字段名的可靠方法**：当 BillQuery 报"元数据中标识为 XXX 的字段不存在"时，改用 `View` 接口查看一条真实单据的完整 JSON 结构，直接看字段真实拼写，比反复猜测大小写变体更快。
